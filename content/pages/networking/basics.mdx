# Networking Basics

The concepts here sit underneath everything in a homelab — routers, containers, VMs, and DNS all rely on these building blocks.

---

## The OSI Model

Network communication happens in 7 layers. Each layer has a specific job and talks only to the layers directly above and below it.

```
Layer 7: APPLICATION
├─ What: Web browsers, email, chat, databases
├─ Protocol: HTTP, HTTPS, SSH, DNS, FTP
└─ Example: OpenWebUI (web interface)

Layer 6: PRESENTATION
├─ What: Encryption, compression, formatting
└─ Example: TLS (HTTPS uses TLS)

Layer 5: SESSION
├─ What: Manages conversation flow
└─ Example: WebSocket connections

Layer 4: TRANSPORT
├─ What: Ensures data arrives correctly
├─ Protocol: TCP (reliable), UDP (fast)
└─ Example: TCP port 80 (HTTP), 443 (HTTPS)

Layer 3: NETWORK
├─ What: Routes data across networks
├─ Protocol: IP (v4, v6)
└─ Example: 192.168.1.20 (Proxmox IP)

Layer 2: DATA LINK
├─ What: Moves data between devices on same network
├─ Protocol: Ethernet, WiFi
└─ Example: MAC address (physical hardware address)

Layer 1: PHYSICAL
├─ What: Cables, radio waves, electricity
└─ Example: Cat6 ethernet cable, WiFi antenna
```

### Layers working together — a real example

```
You open browser: http://192.168.1.20:8080

→ Sending
Layer 7: Browser sends GET /api/chat
Layer 6: Data compressed/formatted
Layer 5: WebSocket session established
Layer 4: TCP stream created
Layer 3: IP header added — from=192.168.1.100, to=192.168.1.20
Layer 2: Ethernet frame built, MAC addresses added
Layer 1: Electrical signals sent through cable

→ Receiving (server side, reversed)
Layer 1: Signals → bits
Layer 2: Frame unpacked
Layer 3: IP packet routed to Proxmox
Layer 4: TCP stream assembled
Layer 7: nginx processes the request
```

## TCP/IP Fundamentals

### IP Addresses

```
IPv4 Format: 192.168.1.20
├─ 4 octets, each 0–255
├─ Total: 4.3 billion addresses (now running out)

Private ranges (not routable on the internet):
├─ 192.168.0.0 – 192.168.255.255  ← your homelab network
├─ 10.0.0.0 – 10.255.255.255
└─ 172.16.0.0 – 172.31.255.255

Public IP: assigned by your ISP, unique on the internet
└─ Changes on router reboot unless you pay for a static IP
```

### TCP — Reliable delivery

TCP (Transmission Control Protocol) guarantees that data arrives in order and nothing is lost. It uses a 3-way handshake to establish a connection before any data flows.

```
Client                  Server
  │── SYN ──────────────→ │    "I want to connect"
  │← SYN-ACK ────────────│    "OK, ready"
  │── ACK ──────────────→ │    "Great, starting"
  │                        │
  │═══ data flowing ══════│
```

Used by: HTTP, HTTPS, SSH, databases — anything where losing a packet is not acceptable.

### UDP — Fast delivery

UDP (User Datagram Protocol) sends packets with no handshake and no retransmission. Faster, but some packets may be lost or arrive out of order.

Used by: DNS queries, video streaming, VoIP, gaming — where speed matters more than perfection.

### TCP vs UDP at a glance

| | TCP | UDP |
|---|-----|-----|
| Guaranteed delivery | ✅ Yes | ❌ No |
| Ordered arrival | ✅ Yes | ❌ No |
| Speed | Slower (handshake overhead) | Faster |
| Use cases | HTTP, SSH, FTP | DNS, video, gaming |

## How Your Router Works

The router is the gateway between your private local network and the internet. It does several jobs simultaneously.

```
Internet ←→ [ASUS Router] ←→ Your Devices
               │
               ├─ DHCP Server: assigns IP addresses to devices
               ├─ NAT: translates private ↔ public IPs
               ├─ Firewall: blocks unauthorized inbound traffic
               ├─ WiFi: wireless access
               └─ DNS Relay: forwards DNS queries upstream
```

### DHCP — Automatic IP assignment

Without DHCP you'd have to manually assign a unique IP to every device. With DHCP, the router handles it:

```
New device connects → Router assigns an IP from its pool
Device disconnects  → IP gets recycled for reuse
Static assignment   → Router always gives the same IP to a known MAC address
```

DHCP reservations (sometimes called "static DHCP") are the best of both worlds — deterministic IPs without manual configuration on every device.

### NAT — Sharing one public IP

Your router has one public IP from your ISP, but many devices behind it. NAT (Network Address Translation) makes this work:

```
Your PC (192.168.1.100:12345)
         ↓ router translates outbound source
Internet sees: 203.0.113.45:60000  (router's public IP)
         ↓ router remembers the mapping
Response arrives at: 203.0.113.45:60000
         ↓ router translates back
Your PC receives at: 192.168.1.100:12345
```

Side effects of NAT:
- Your private IPs are never exposed to the internet ✅
- Inbound connections from the internet are blocked by default ✅
- Port forwarding punches a specific hole through for services you want to expose

### SPI Firewall

Stateful Packet Inspection tracks conversations, not just ports:

```
Outbound request:  Your PC → google.com:443
Router notes:      "Opened port, expecting response"
Inbound response:  google.com → your PC
Router checks:     "This matches an open conversation — allow"

Unsolicited inbound (attack):
Router checks:     "I don't recognise this — drop"
```

## IP Addressing

### Subnet notation (`/24`)

```
192.168.1.0/24

/24 means the first 24 bits are the network, last 8 are hosts.
Usable range: 192.168.1.1 – 192.168.1.254  (254 hosts)

.0   = network address (not assignable)
.1   = typically the router/gateway
.255 = broadcast address (not assignable)
```

### Example homelab address scheme

```
192.168.1.0/24

.1    ASUS Router (gateway)
.20   Proxmox host
.25   OpenWebUI container
.26   Ollama container
.30   NAS / backup storage
.50–.254  DHCP pool (dynamic devices)
```

### IPv6

IPv6 uses 128-bit addresses (vs IPv4's 32 bits), solving the address exhaustion problem. Format: `2001:0db8:85a3::8a2e:0370:7334`. Not required for a homelab today — dual-stack (both IPv4 and IPv6 active) is the transition path when needed.

## DNS Resolution

DNS (Domain Name System) converts human-readable names into IP addresses.

```
Without DNS:
  You type:  http://openwebui.local:8080
  Computer:  "I don't know the IP for that"

With DNS:
  You type:  http://openwebui.local:8080
  Computer → DNS server: "What is openwebui.local?"
  DNS → Computer: "192.168.1.25"
  Computer → 192.168.1.25:8080: request
  OpenWebUI loads ✅
```

### DNS resolution chain

```
Your device → Router (local DNS relay)
                  ↓ if not cached
              Upstream DNS (1.1.1.1 Cloudflare or 8.8.8.8 Google)
                  ↓
              Authoritative DNS server for that domain
                  ↓
              IP address returned, cached by router
```

### Local DNS names (`*.local`)

mDNS (Multicast DNS) lets devices on the same network discover each other by hostname without any configuration:

```
Container sets hostname: openwebui
mDNS broadcasts:         "I am openwebui.local at 192.168.1.25"
Other devices can use:   http://openwebui.local  (no IP needed)
```

### Custom local DNS via dnsmasq

Routers running dnsmasq (including ASUS Merlin) can be given static hostname → IP mappings:

```
address=/proxmox.home/192.168.1.20
address=/nas.home/192.168.1.30
```

This is more reliable than mDNS for server infrastructure.
