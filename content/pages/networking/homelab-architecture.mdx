# Homelab Network Architecture

This page covers how a Proxmox-based homelab network is structured — from physical hardware down to individual containers — along with common failure modes and how to diagnose them.

---

## Physical Layout

```
Internet
    │
    ↓
[ASUS RT-AX88U Router]  192.168.1.1
├─ LAN1: [Proxmox Host]          192.168.1.20  (ethernet, static)
├─ LAN2: [NAS / Backup Storage]  192.168.1.30  (ethernet, static)
├─ LAN3–8: [Available]
│
WiFi: computers, phones, guests
```

### Network zones

```
Zone 1 — WAN
└─ Internet ↔ Router
   └─ ISP provides public IP (changes on reboot unless static)

Zone 2 — LAN (192.168.1.x)
├─ Proxmox host: .20
├─ Backup storage: .30
└─ All internal communication happens here

Zone 3 — Guest WiFi (192.168.2.x, isolated)
├─ Cannot reach: Proxmox, storage, internal servers
└─ Can reach: Internet only

Zone 4 — Container network (inside Proxmox)
├─ OpenWebUI container: .25
├─ Ollama container: .26
└─ Managed by Linux bridges inside Proxmox
```

## Traffic Flow Examples

### Access OpenWebUI from your PC

```
Your PC (192.168.1.100)
     ↓ < 1 ms  (WiFi or ethernet to router)
ASUS Router (192.168.1.1)
     ↓ < 1 ms  (ethernet)
Proxmox Host (192.168.1.20)
     ↓ < 1 ms  (Linux bridge — same machine)
OpenWebUI Container (192.168.1.25)
     ↓
Response flows back the same path

Total round-trip: < 5 ms  (imperceptible)
```

### Container accesses an external service

```
OpenWebUI Container → Proxmox Host (has routes)
     ↓
ASUS Router (NAT translates private IP → router's public IP)
     ↓
Internet
     ↓
Response comes back to router's public IP
Router translates back to 192.168.1.25
Container receives response

The container's private IP is never visible on the internet.
```

## Proxmox Networking Layer

### Physical to virtual

```
Physical hardware:
└─ Ethernet card: enp5s0 (1 Gbps)
   └─ Connected to: ASUS Router (LAN port)

Proxmox creates:
└─ Linux bridge: vmbr0
   ├─ Connects: physical card + containers + VMs
   └─ IP: 192.168.1.20  (Proxmox management address)
```

### Linux bridges explained

Think of a bridge as a virtual network switch inside the Proxmox host.

```
Without bridge:
├─ Physical card: can reach external network only
├─ Containers: no network
└─ VMs: no network

With vmbr0 bridge:
Internet ←→ ASUS Router ←→ [eth: enp5s0] ←→ vmbr0
                                              ├─ Proxmox host
                                              ├─ OpenWebUI container
                                              ├─ Ollama container
                                              └─ VMs

All connected to the same bridge can talk to each other
and to the outside world through the physical card.
```

### Multiple bridges (future)

You can create additional bridges to isolate traffic:

```
vmbr0 (Main — 192.168.1.x):
├─ Proxmox host: .20
├─ OpenWebUI: .25
└─ Storage: .30

vmbr1 (Guest — 192.168.2.x):
└─ Guest VMs (isolated from vmbr0 by firewall rules)
```

## Virtual Bridges vs VLANs

### Bridge (what you have now)

All devices on the same bridge are on the same flat L2 network — they can all reach each other directly.

```
vmbr0 — everyone on 192.168.1.x can talk to everyone else
```

### VLANs (logical isolation)

VLANs partition a single physical switch into isolated logical networks. Devices on different VLANs cannot communicate without going through a router/firewall.

```
VLAN 10 — Internal (192.168.10.x)
├─ Proxmox, containers, storage
└─ Cannot be reached from VLAN 20

VLAN 20 — Guest (192.168.20.x)
├─ Guest machines
└─ Cannot reach VLAN 10

Requires: a managed switch that supports 802.1q VLAN tagging
```

VLANs are not needed for a single-host homelab initially — a flat bridge is simpler and works well.

## Container Networking (LXC)

LXC containers share the host's Linux kernel but get their own isolated network namespace.

```
Container has:
├─ Virtual ethernet interface: eth0
├─ IP address: 192.168.1.25 (assigned by DHCP from router)
└─ Gateway: 192.168.1.1 (ASUS Router)

Connection path:
Container eth0 ←→ veth pair ←→ vmbr0 bridge ←→ physical ethernet
```

The container looks and behaves like a physical machine on the network. Other devices on `192.168.1.x` can reach it directly.

## VM Networking (KVM)

VMs get a fully emulated virtual network card (virtio for best performance). The connection path is the same as containers — VM virtual NIC → vmbr0 → physical card.

### Performance comparison

```
Native Proxmox host ping to router:   < 1 ms
Container ping to router:             < 1 ms
VM ping to router:                    < 1 ms
```

In practice there is no noticeable difference for homelab workloads.

## Network Security

### Layered defence

```
Layer 1 — ASUS Router (perimeter)
├─ NAT: hides private network
├─ SPI firewall: drops unsolicited inbound traffic
└─ Port forwarding: only explicitly opened ports are reachable

Layer 2 — Proxmox host (iptables)
├─ Firewall rules per bridge
└─ Can block traffic between containers if needed

Layer 3 — Container / service
└─ Application-level auth (passwords, API keys, mTLS)
```

### Firewall rule example (iptables)

```bash
# Allow web traffic to OpenWebUI from LAN only
iptables -I INPUT -s 192.168.1.0/24 -p tcp --dport 8080 -j ACCEPT
iptables -I INPUT -p tcp --dport 8080 -j DROP

# Block Telnet inbound
iptables -I INPUT -i eth0 -p tcp --dport 23 -j DROP
```

### Pi-hole (DNS-level ad/malware blocking)

```
Without Pi-hole:
  Device → DNS → malware.com resolves → device connects → infected

With Pi-hole:
  Device → Pi-hole DNS → "malware.com is blocked" → drops query → safe

Benefits:
├─ Network-wide (all devices protected automatically)
├─ No per-device config
├─ Blocks ads and malicious domains
└─ Free, open-source
```

## Troubleshooting

### Container can't reach the internet

```bash
# Step 1: Test raw connectivity
ping 8.8.8.8
# Fails → networking issue. Works → DNS issue (see below)

# Step 2: Check bridge
ip addr show
# Should show a veth interface connected to vmbr0

# Step 3: Check default gateway
ip route
# Should show: default via 192.168.1.1

# Step 4: Restart networking
systemctl restart networking
```

### Container can ping but DNS fails

```bash
# Check DNS config
cat /etc/resolv.conf
# Should show: nameserver 192.168.1.1

# Test DNS directly
nslookup google.com 8.8.8.8
# Works → router DNS is the problem; set upstream DNS to 8.8.8.8 in router UI

# Add fallback manually
echo "nameserver 8.8.8.8" >> /etc/resolv.conf
```

### Can't reach container from another PC on the LAN

```bash
# From your PC: verify you're on the same subnet
ipconfig      # Windows
ifconfig      # macOS / Linux
# Should show: 192.168.1.x

# From Proxmox host: verify routing
route -n
# Should show: 192.168.1.0 via vmbr0

# Temporarily disable firewall to test
iptables -F   # flushes all rules (testing only — re-enable after)

# Check container is actually listening
ss -tlnp | grep 8080   # run inside the container
```

### Two containers can't reach each other

```bash
# Both containers must be on the same bridge
# Check in Proxmox UI: Container > Network > Bridge field

# From container A, ping container B
ping 192.168.1.26

# Verify target service is running and listening
ss -tlnp | grep 11434  # Ollama default port
```

### Slow network performance

```bash
# Run a speed test inside container
iperf3 -c 192.168.1.1
# Expected: 900+ Mbps on gigabit ethernet

# Check MTU
ip link show
# Should be 1500 bytes

# Check host CPU (could be the bottleneck)
top   # run on Proxmox host; CPU usage > 80% = host is saturated
```
